Modèle de lecture audio — quelle approche ?

Tension : fluidité des transitions DJ ↔ navigation claire par track

A

Fichier unique + seekTo(ms) — actuel

liveset.mp3 ──────────────────────────────
00:00     07:23     14:51     22:04
▲Track1  ▲Track2  ▲Track3  ▲Track4
seekTo(443000) → continue lecture naturellement

Pour

  • Transitions intactes
  • Simple
  • Seek local = 0ms lag

Contre

  • Cliquer "Track 3" = démarrage à froid dans le mix
  • Pas d'aperçu isolé d'un track
B

Pre-roll seek — "Soft Entry" ★ Recommandé

liveset.mp3 ──────────────────────────────
          07:23
          ▲Track2
User clique Track2  → seek 06:53 (30s avant)
          └─ entend la fin du mix-in de T1→T2 → T2 complet → T2→T3

Pour

  • Transitions entendues
  • Contexte musical préservé
  • 0 code supplémentaire côté ExoPlayer
  • pre_roll configurable (15s, 30s…)

Contre

  • Le "début" du track inclut la fin du précédent
C

HLS local via ffmpeg — "Virtual Tracks"

liveset.mp3
↓ ffmpeg (1x, offline, après download)
playlist.m3u8
segment_001.ts 00:00→07:53 (+30s overlap)
segment_002.ts 06:53→15:21 (±30s overlap)
segment_003.ts 14:21→22:34 (±30s overlap)
ExoPlayer joue HLS natif → transition seamless, seek par segment

Pour

  • Chaque "track" est un segment discret
  • ExoPlayer HLS natif = gapless
  • Overlap = transition préservée

Contre

  • ffmpeg à lancer sur VM après download
  • Stockage × 1.1 (overlaps)
  • Complexité pipeline
Note sur les transitions DJ
Dans un mix technique, le "début" d'un track n'est pas un instant précis — c'est une zone de 20-90s où deux tracks coexistent. L'option B préserve cette réalité sans infrastructure supplémentaire.
